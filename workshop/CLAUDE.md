# Workshop CLAUDE.md

Development guidance for building the TestDevLab AI Bootcamp platform. All Claude Code sessions working on `workshop/` must follow these rules.

## Architecture Overview

This is a **fully automated AI bootcamp platform** with three pillars:

1. **Curriculum** (`sessions/`) — 10 session guides teaching Claude Code features via building a React app
2. **Platform** (`platform/`, `hooks/`, `gates/`, `scoring/`) — Registration, prompt logging, phase gates, automated scoring
3. **Admin** (`admin/`) — Instructor dashboard, cohort management, score exports

## Critical Rules

### No Copy-Paste Prompts

Session guides MUST NOT contain exact prompts students can copy-paste. Instead provide:

- Clear goals and requirements ("Build a test case CRUD page")
- Constraints and rules ("Must use ISTQB severity levels")
- Acceptance criteria ("The page must show create/read/update/delete operations")
- Hints that guide thinking ("Consider what data model Claude needs to know about")
- Links to reference modules for feature documentation

**WRONG**: "Type this prompt: Create a React component that shows test cases in a table"
**RIGHT**: "Build a page that displays test cases. Think about what columns a QA engineer needs to see at a glance. Your CLAUDE.md standards should guide the data model."

### Progressive Dependencies

Features created in early sessions MUST be used in later sessions. The gate validators enforce this:

- S1 slash commands -> S7 plugin must bundle them
- S2 CLAUDE.md rules -> S6 hooks enforce them
- S3 skills -> S4 agents reference them, S7 plugin bundles them
- S4 agents -> S7 plugin bundles them
- S5 MCP configs -> S10 CI uses them
- S6 hooks -> S7 plugin bundles them

When writing gate validators, always check for cross-session references.

### Session Guide Structure

Every session README.md follows this exact structure:

```
# Session N: [Title] -- [Feature Name]
## Learning Objectives (3-5 bullets)
## Prerequisites (previous gate + tools needed)
## Schedule (time table)
## The Feature (explanation + link to reference module)
## What You Will Build (specific deliverables)
## Requirements
### Must-Have (gate checks)
### Should-Have (bonus points)
## Rules
## Hints (Not Solutions)
## Jira Integration (sessions 5+)
## Verification (self-check before ./bootcamp complete-session N)
```

### Jira Integration (Sessions 5+)

From Session 5 onward, students must use Atlassian MCP to manage their work in Jira:

- Create epics per feature area
- Write stories with acceptance criteria
- Link dependent tickets
- Update status as work progresses
- All done via Claude Code prompts, NOT manual Jira UI

### Gate Validators

Each `gate_sNN_*.py` must:

- Import from `gates/common.py` for shared utilities
- Return a list of `CheckResult(passed, message, points)` objects
- Include both critical checks (must pass) and bonus checks (extra points)
- Validate progressive dependencies from earlier sessions
- Be deterministic (no AI calls in gates — use file/content checks only)

### Scoring Engine

The scoring system has two modes:

- **AI mode** (default): Uses `claude -p --output-format json` for prompt quality analysis
- **Heuristic mode** (`--no-ai`): Falls back to pattern matching for environments without API access

Rubric weights: prompt_quality=25%, efficiency=15%, deliverable_quality=20%, standards_compliance=15%, final_task=25%

### Hook Scripts

Hook scripts in `hooks/` must:

- Always exit 0 (never block student work, even if logging fails)
- Execute in under 1 second
- Use `mkdir -p` defensively for directory creation
- Use `>>` append mode (never overwrite)
- Read student ID from `~/.claude-bootcamp/student-id`
- Write JSONL to `~/.claude-bootcamp/logs/{student-id}/session-{N}/`

## Tech Stack

### Platform (what we build)

- **Language**: Python 3.10+ (matches existing `scripts/` conventions)
- **Database**: SQLite via stdlib `sqlite3`
- **CLI**: argparse subcommands
- **Hooks**: Shell scripts (bash, consistent with `06-hooks/` examples)
- **Quality**: ruff, mypy, bandit, pytest (same tools as main repo)

### QA Command Center (what students build)

- **Backend**: Node.js + Express
- **Frontend**: React via Vite
- **Database**: SQLite
- **Everything generated by Claude Code** — students prompt, never write code

## Code Style

- Follow existing repo conventions: ruff for linting/formatting, mypy for types
- Python docstrings only on public functions
- Shell scripts: use `set -euo pipefail` header
- All markdown must pass the repo's pre-commit hooks (markdownlint, cross-references)
- Commit format: `feat(workshop): Add session 3 guide` or `fix(workshop/gates): Fix S5 MCP validation`

## Testing

```bash
pytest workshop/tests/ -v
pytest workshop/tests/ -v --cov=workshop --cov-report=html
```

Every platform module must have corresponding tests. Gate validators test against fixture project directories in `tests/fixtures/`.
